# CIM 产品迭代与阶段目标设计

最后整理日期：2026-06-19

## 1. 文档目标

本文件用于把前面的架构设计、功能设计和对象映射设计，进一步落成可以执行的产品迭代方案。

本文件重点回答四个问题：

1. 这个项目应按什么阶段推进。
2. 每个阶段要交付什么能力。
3. 每个阶段的边界在哪里，哪些不做。
4. 什么时候可以认为该阶段完成。

## 2. 产品总目标

`cimdraw` 的总目标不是只支持“打开一份 CIM 文件”，而是逐步形成一套面向电力 CIM/CGMES 的桌面工作台，最终具备下面四条主线能力：

1. `模型导入`
2. `图模映射`
3. `拓扑与分析`
4. `导出与集成`

从产品角度看，建议把整个建设过程拆成 `P0 / P1 / P2 / P3` 四个阶段。

## 3. 总体迭代路线

### 3.1 P0：可读、可看、可定位

目标：

1. 把 `EQ + SSH` 导进来。
2. 把核心 CIM 对象浏览起来。
3. 把核心设备落成图纸初稿。
4. 做到对象和图元双向定位。

这一阶段的关键词是：

1. 可导入
2. 可浏览
3. 可建图
4. 可定位

### 3.2 P1：可校验、可分析、可回写

目标：

1. 基于导入模型跑本地拓扑分析。
2. 把当前拓扑与供电/接地路径能力接到 `CimModel`。
3. 支持部分图纸编辑回写语义模型。
4. 支持基础导出与校验闭环。

这一阶段的关键词是：

1. 可校验
2. 可分析
3. 可回写
4. 可导出

### 3.3 P2：可扩展、可批处理、可对接

目标：

1. 接入 `TP / SV`。
2. 支持批量导入、批量校验、批量转换。
3. 预留或接入外部分析引擎适配器。
4. 增加模型差异比较与 Profile 管理。

这一阶段的关键词是：

1. 可扩展
2. 可批量
3. 可对接
4. 可治理

### 3.4 P3：可平台化

目标：

1. 增加 `DL / GL` 支持。
2. 增强查询缓存或图数据库后端。
3. 完善外部工具链与服务化集成。
4. 形成稳定版本化发布能力。

这一阶段的关键词是：

1. 可平台化
2. 可集成
3. 可治理
4. 可发布

## 4. 各阶段具体目标

### 4.1 P0 阶段目标

#### 范围

必须完成：

1. `EQ + SSH` 导入
2. 内部 `CimModel`
3. 对象树浏览
4. 断路器、刀闸、母线、变压器、负荷、发电机的图元映射
5. 对象与图元双向定位
6. 导入日志与问题初稿

明确不做：

1. 全量 CGMES 类支持
2. 完整导出
3. 外部分析引擎接入
4. DL / GL 支持
5. 复杂自动修复

#### 阶段交付物

1. `src/cim/model`
2. `src/cim/importer`
3. `src/cim/mapping`
4. `CimModelBrowserDock`
5. `CimImportExportWizard` 的导入链路
6. 基础导入测试数据与测试用例

#### 完成标志

1. 能导入一组最小 `EQ + SSH` 数据
2. 能在对象树中看到核心对象
3. 能生成一版基础单线图初稿
4. 能从对象定位到图元、从图元定位到对象

### 4.2 P1 阶段目标

#### 范围

必须完成：

1. `CimTopologyMapper`
2. `CimValidationEngine`
3. 本地拓扑分析接入
4. 供电路径、接地路径、开关预演联动
5. 基础图纸编辑回写
6. 最小可用导出子集

明确不做：

1. 全套外部求解器集成
2. 完整状态估计与短路真实求解
3. 地理视图

#### 阶段交付物

1. `src/cim/validation`
2. `src/cim/exporter`
3. `CimValidationDock`
4. `CimAnalysisDock`
5. `CimModel -> PowerTopologyAnalysisSnapshot` 转换链路

#### 完成标志

1. 能对导入模型执行基础结构校验
2. 能复用现有拓扑能力跑导通和路径分析
3. 能把部分图元编辑结果回写到 `CimModel`
4. 能导出最小可用的模型子集和映射报告

### 4.3 P2 阶段目标

#### 范围

必须完成：

1. `TP / SV` 支持
2. 批量导入/校验/转换
3. 差异比较
4. Profile 管理
5. 外部分析适配器接口稳定化

#### 阶段交付物

1. `src/cim/profile`
2. `src/analysis/adapters`
3. 批处理命令入口
4. 差异比较报告

#### 完成标志

1. 能批量导入多个数据集
2. 能生成批量校验结果
3. 能比较两版模型差异
4. 能把分析适配器接到统一接口上

### 4.4 P3 阶段目标

#### 范围

必须完成：

1. `DL / GL` 支持
2. 查询缓存增强
3. 服务化转换或校验对接
4. 发布与版本治理

#### 阶段交付物

1. `src/cim/store` 增强实现
2. 查询后端或图数据库适配层
3. 发布说明和升级说明

#### 完成标志

1. 能处理布局与地理补充信息
2. 能支撑较大模型查询和浏览
3. 能稳定对接外部工具链

## 5. 各阶段模块拆分建议

### 5.1 P0 模块优先级

P0 必做模块：

1. `CimModel`
2. `CgmesPackageImporter`
3. `CimGraphicMapper`
4. `CimIdIndex`
5. `CimModelBrowserDock`

P0 可延后模块：

1. `CimValidationEngine`
2. `CimTopologyMapper`
3. `ICimExporter`
4. `IPowerAnalysisAdapter`

### 5.2 P1 模块优先级

P1 必做模块：

1. `CimTopologyMapper`
2. `CimValidationEngine`
3. `ICimExporter`
4. `NativeTopologyAdapter`

### 5.3 P2 模块优先级

P2 必做模块：

1. `CimProfileSet`
2. `PowerGridModelAdapter` 或统一外接适配接口
3. `CimModelDiffService`
4. `BatchImportService`

## 6. 阶段风险与控制点

### 6.1 P0 风险

1. 直接把 CIM 类做成过重继承树，导致推进变慢。
2. 过早追求全量类支持，导致导入器迟迟不可用。
3. 图模映射过度依赖当前图元实现细节，后续扩展困难。

控制建议：

1. 统一 `CimObject` 模型先落地。
2. 先支持最小对象集，不追全。
3. 先做单向导入和建图，再做复杂回写。

### 6.2 P1 风险

1. 回写逻辑和图形编辑耦合过深。
2. 拓扑结果和导入 `TP` 不一致时难以解释。

控制建议：

1. 区分导入拓扑和本地重建拓扑。
2. 回写先覆盖少数关键对象，不一开始做全量双向同步。

### 6.3 P2 / P3 风险

1. 外部分析工具接口不稳定。
2. 模型规模上来后查询性能不足。

控制建议：

1. 统一适配器接口。
2. 把缓存和图数据库能力放在后续阶段独立演进。

## 7. 阶段验收方式

### 7.1 P0 验收

1. 给定一组最小 `EQ + SSH` 数据，可以导入。
2. 能生成可见图纸。
3. 能定位断路器、刀闸、母线、变压器、负荷、发电机。

### 7.2 P1 验收

1. 能运行基础校验。
2. 能跑本地拓扑与路径分析。
3. 能回写部分编辑结果。

### 7.3 P2 验收

1. 能批量处理多组模型。
2. 能输出差异和批量校验结果。
3. 能接统一分析适配器。

### 7.4 P3 验收

1. 能处理布局和地理补充信息。
2. 能在较大模型上稳定浏览和查询。

## 8. 结论

这份迭代设计的核心思想是：

1. 先把最小闭环跑通。
2. 再把校验、分析和回写接上。
3. 最后再做批处理、外接引擎和平台化能力。

这样可以最大化复用当前 `cimdraw` 的代码基础，同时避免项目一开始就陷入“标准太大、范围太散、迟迟不可交付”的风险。
