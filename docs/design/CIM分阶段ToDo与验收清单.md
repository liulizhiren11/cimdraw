# CIM 分阶段 ToDo 与验收清单

最后整理日期：2026-06-29

## 1. 文档目标

本文件用于把阶段设计进一步压缩成一份可直接执行的里程碑清单。

补充说明：

1. 本次已按当前仓库中的 `src/cim/*`、`tests/test_cim_*`、`tests/data/cim/*` 实际实现情况补充 `√` 标记。
2. `P0` 中仅对当前代码和自动化测试可以直接证明的事项补 `√`，其余仍保留待办状态。

适用场景：

1. 研发排期
2. 周迭代拆任务
3. 提测前自检
4. 里程碑验收
5. 阶段复盘

## 2. 使用方式

建议每个阶段都按下面四步使用：

1. 先确认阶段目标和边界。
2. 再按 `todoList` 落开发任务。
3. 按测试标准完成验证。
4. 按验收方式判断是否允许进入下一阶段。

补充约定：

1. 每个阶段都必须同时给出“成功样例、失败样例、演示路径”。
2. 验收结论必须可追溯到测试记录、日志或截图，不能只写“已验证”。
3. 若存在已知问题，必须明确是否阻断进入下一阶段。

## 3. P0 阶段清单

### 3.1 阶段目标

P0 的目标是打通 `EQ + SSH -> 对象树 -> 图纸初稿 -> 双向定位` 这条最小闭环。

### 3.2 进入验收前置条件

1. `P0` 范围内代码已合入可构建分支。
2. 最小成功样例、失败样例、核心设备覆盖样例都已固定。
3. 对应自动化测试可重复执行，且结果可导出。

### 3.3 ToDo List

- √ 定义 `CimObject / CimReference / CimModel / CimProfileSet`
- √ 建立 `src/cim/importer/`
- √ 定义 `ICimImporter`
- √ 实现 `CgmesPackageImporter`
- √ 支持最小 `EQ + SSH` 数据读取
- √ 输出导入日志、错误列表、对象统计

#### 图模映射

- √ 建立 `src/cim/mapping/`
- √ 实现 `CimIdIndex`
- √ 实现 `CimGraphicMapper`
- √ 覆盖 `Breaker / Disconnector / BusbarSection / PowerTransformer / EnergyConsumer / SynchronousMachine`
- [ ] 建立对象和图元双向索引

#### 界面与交互

- √ 增加 `导入 CIM/CGMES` 菜单入口
- √ 增加 `CimModelBrowserDock`
- √ 支持对象树搜索
- √ 支持对象定位图元
- √ 支持图元定位对象

#### 测试数据

- √ 准备最小 `EQ + SSH` 样例
- √ 准备核心设备覆盖样例
- √ 准备导入失败样例

### 3.4 测试标准

#### 单元测试标准

- `CimModel` 能按 `mRID` 查询对象
- `CimModel` 能返回对象总数和类分布
- `CimIdIndex` 能做双向索引
- `CimGraphicMapper` 能映射到正确图元类型

#### 组件测试标准

- 导入器能读取最小 `EQ + SSH` 目录样例
- 导入器对缺失文件和空文件能给出明确错误
- 导入器能输出导入统计
- 映射器能为核心设备生成图元定义

#### 集成测试标准

- 导入数据后能生成 `CimModel`
- 对象树中能显示核心对象
- 能生成基础单线图初稿
- 能完成对象与图元双向定位

#### 放行红线

- 导入成功样例仍存在崩溃或静默失败
- 失败样例没有明确错误提示
- 双向定位只能单向成立，另一向失效

### 3.5 验收方式

#### 验收输入

1. 一组最小 `EQ + SSH` 样例
2. 一组导入失败样例
3. 一组覆盖核心设备的展示样例

#### 验收步骤

1. 从菜单执行 `导入 CIM/CGMES`
2. 导入最小样例并查看导入日志
3. 打开对象树核对对象是否完整显示
4. 打开图纸核对核心设备是否生成
5. 逐个验证对象到图元、图元到对象的定位
6. 导入失败样例并检查错误提示

#### 通过判定

1. 导入成功样例无崩溃、无静默失败
2. 对象树和图纸都能看到核心设备
3. 双向定位可用
4. 失败样例有可读错误信息
5. 自动化测试和手工演示结论一致

#### 验收输出

1. 演示录像或截图
2. 导入日志样例
3. `P0` 测试通过记录
4. `P0` 已知问题清单
5. 是否进入 `P1` 的结论

## 4. P1 阶段清单

### 4.1 阶段目标

P1 的目标是把 `P0` 的“可看”提升到“可校验、可分析、可回写、可最小导出”。

### 4.2 进入验收前置条件

1. `P0` 主链路已稳定，不再依赖临时手工修复。
2. 校验、分析、回写、导出各自至少有一组固定样例。
3. 回写和导出结果具备可核对的对比基线。

### 4.3 ToDo List

#### 校验

- [ ] 建立 `src/cim/validation/`
- [ ] 定义 `CimValidationIssue`
- [ ] 实现结构校验
- [ ] 实现引用完整性校验
- [ ] 实现图模映射完整性校验
- [ ] 增加 `CimValidationDock`

#### 拓扑与分析

- [ ] 实现 `CimTopologyMapper`
- [ ] 打通 `CimModel -> PowerTopologyAnalysisSnapshot`
- [ ] 接通导通分析
- [ ] 接通供电路径分析
- [ ] 接通接地路径分析
- [ ] 接通开关预演
- [ ] 增加分析结果面板

#### 回写与导出

- [ ] 实现关键图元属性回写
- [ ] 建立 `src/cim/exporter/`
- [ ] 定义 `ICimExporter`
- [ ] 导出最小可用对象子集
- [ ] 输出映射报告

### 4.4 测试标准

#### 单元测试标准

- `CimValidationEngine` 能输出错误、警告和建议
- `CimTopologyMapper` 能生成稳定拓扑快照
- 导出器能导出最小对象子集

#### 组件测试标准

- 校验引擎能识别缺失引用和非法结构
- 拓扑映射能识别端子连接关系
- 回写模块能同步关键语义字段
- 导出器能输出导出报告

#### 集成测试标准

- 导入后可运行结构校验
- 可运行导通、供电路径、接地路径分析
- 修改图元后能回写部分语义对象
- 可导出最小模型子集

#### 放行红线

- 分析结果与现有拓扑能力在关键样例上明显不一致且无法解释
- 回写会破坏模型基础结构
- 导出结果无法被重新核对或缺少导出报告

### 4.5 验收方式

#### 验收输入

1. 一组可跑拓扑分析的中等样例
2. 一组包含结构问题的异常样例
3. 一组可导出的最小样例

#### 验收步骤

1. 导入样例并运行模型校验
2. 记录错误、警告和建议输出
3. 运行导通、供电路径、接地路径分析
4. 手动修改关键图元属性并触发回写
5. 导出最小模型子集并检查报告

#### 通过判定

1. 校验、分析、回写、导出四条链路全部可运行
2. 分析结果与现有本地拓扑能力在关键样例上结论一致
3. 回写后的模型仍可通过基础校验
4. 导出结果有报告且可用于人工复核
5. 至少一组样例完成全链路串行演示

#### 验收输出

1. 校验问题清单
2. 分析结果截图或记录
3. 回写前后对比说明
4. 导出样例与导出报告
5. `P1` 已知问题清单
6. 是否进入 `P2` 的结论

## 5. P2 阶段清单

### 5.1 阶段目标

P2 的目标是把单次交互能力提升为可批处理、可差异比较、可统一对接外部分析。

### 5.2 进入验收前置条件

1. 批量处理入口、差异比较入口、适配器入口均已稳定命名。
2. 多模型样例、差异样例、适配器异常样例已固定。
3. 汇总报告格式已冻结，便于横向对比。

### 5.3 ToDo List

#### 批处理

- [ ] 增加命令行批量导入入口
- [ ] 增加批量校验入口
- [ ] 增加批量转换入口
- [ ] 增加汇总报告输出

#### Profile 与差异

- [ ] 建立 `src/cim/profile/`
- [ ] 支持基础 Profile 管理
- [ ] 支持模型差异比较
- [ ] 输出差异报告

#### 外接分析

- [ ] 建立 `src/analysis/adapters/`
- [ ] 定义 `IPowerAnalysisAdapter`
- [ ] 实现 `NativeTopologyAdapter`
- [ ] 预留 `PowerGridModelAdapter`
- [ ] 统一任务输入输出口径

### 5.4 测试标准

#### 单元测试标准

- 差异比较服务能识别新增、删除、修改
- 批处理服务能稳定组织任务输入输出
- 适配器接口满足统一协议

#### 组件测试标准

- 批量导入多个样例时状态不会互相污染
- 批量校验可输出汇总结果
- 差异报告格式稳定
- 适配器异常能被统一捕获

#### 集成测试标准

- 可批量处理多组模型
- 可输出批量校验报告
- 可比较两版模型差异
- 可接统一分析适配器接口

#### 放行红线

- 批处理结果受前一个任务残留状态影响
- 差异报告不可追踪到具体对象
- 适配器失败会拖垮整个任务批次

### 5.5 验收方式

#### 验收输入

1. 多组批量导入样例
2. 两版存在差异的同源模型
3. 一组适配器正常样例和异常样例

#### 验收步骤

1. 执行批量导入、批量校验、批量转换
2. 检查批处理汇总报告
3. 执行差异比较并人工抽样核对
4. 通过统一适配器入口执行分析
5. 模拟适配器失败并检查错误记录

#### 通过判定

1. 批量任务可重复执行且结果稳定
2. 差异比较结果与人工核对一致
3. 成功与失败场景都能按统一口径输出结果
4. 单个任务失败不会导致整批结果不可用
5. 汇总报告可作为阶段归档材料直接使用

#### 验收输出

1. 批处理汇总报告
2. 差异比较报告
3. 适配器接入说明
4. 异常日志样例
5. `P2` 已知问题清单
6. 是否进入 `P3` 的结论

### 6.2 进入验收前置条件

1. `DL / GL` 样例、大模型性能样例、外部工具链接口样例已准备完成。
2. 性能阈值、恢复时限、版本兼容范围已提前约定。
3. 发布与升级资料模板已形成统一格式。

### 6.3 ToDo List

- [ ] 支持 `GL`
- [ ] 支持布局回读
- [ ] 支持布局回写
- [ ] 建立图纸和地理对象关联

#### 查询与性能

- [ ] 形成版本兼容说明
- [ ] 形成样例数据治理规范

### 6.4 测试标准

#### 单元测试标准

- `DL / GL` 对象可正确解析
- 查询缓存命中和失效逻辑可验证
- 大模型分页查询结果稳定

#### 组件测试标准

- 布局信息导入导出前后一致
- 查询缓存更新不会造成脏读
- 外部查询或存储适配层返回结果稳定

#### 集成测试标准

- 可处理 `DL / GL` 数据集
- 可稳定浏览较大模型
- 可稳定连接外部工具链

#### 放行红线

- 含 `DL / GL` 的主样例仍需人工补数据才能使用
- 大模型性能不达标且没有缓解方案
- 外部工具链异常时系统无法恢复

### 6.5 验收方式

#### 验收输入

1. 含 `DL / GL` 的完整样例
2. 一组较大模型性能样例
3. 一组外部工具链联调样例

#### 验收步骤

1. 导入含布局与地理信息的数据集
2. 核对图纸布局和地理对象关联
3. 执行大模型浏览、搜索、定位
4. 联调外部工具链并回传结果
5. 核对发布、升级、兼容文档

#### 通过判定

1. `DL / GL` 支持可用
2. 大模型浏览和查询性能达到预期
3. 外部集成稳定可恢复
4. 发布和升级资料完整
5. 至少一次完整交付演练通过

#### 验收输出

1. 布局与地理样例
2. 性能测试记录
3. 联调记录
4. 发布说明和升级说明
5. `P3` 版本放行结论
