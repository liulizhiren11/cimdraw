# CIM 开发 ToDo 与测试方案

最后整理日期：2026-06-29

## 1. 文档目标

本文件用于把前面的架构设计、功能设计和阶段设计，进一步落成研发侧可以直接执行的工作清单与测试方案。

说明：

1. 本文件保留的是早期分阶段研发拆解口径，包含当时的历史待办项。
2. 当前仓库真实完成状态请以《CIM标准图元补齐执行ToDoList.md》和《CIM标准模型与工程图元对应表.md》为准。
3. 如果本文件中的旧待办与当前实现状态不一致，应以后两份文档和对应自动化测试结果为准。
4. 本次已按当前仓库中的 `src/cim/*`、`tests/test_cim_*`、`tests/data/cim/*` 实际实现情况补充 `√` 标记。

本文件重点回答四个问题：

1. 每个阶段优先做什么。
2. 每类能力需要拆成哪些开发任务。
3. 每个阶段重点测什么。
4. 如何把测试结果转成阶段放行依据。

## 2. 使用原则

建议按下面顺序使用本文件：

1. 先看当前阶段范围，只做当前阶段必须项。
2. 再按 `todoList` 拆成开发任务、联调任务、测试任务。
3. 每完成一个子模块，就补对应单元测试或组件测试。
4. 阶段末按集成测试、回归测试和验收测试统一收口。

## 3. 阶段执行模板

建议每个阶段都按同一套节奏推进，避免“功能做完了，但提测材料不齐”：

1. `需求收口`
   明确本阶段必须完成、明确不做、可顺延项。
2. `开发拆分`
   把任务拆成模型、导入、映射、UI、测试数据、测试用例六类。
3. `用例先行`
   先准备最小成功样例、边界样例、失败样例，再补实现。
4. `联调收口`
   以一条完整链路作为里程碑，不接受“模块都写了但链路没跑通”。
5. `提测归档`
   输出测试记录、已知问题、验收截图或日志。

## 5. 测试策略总览

建议测试分成六层：

1. 单元测试：验证类和函数级行为正确。
2. 组件测试：验证导入器、映射器、校验器等模块边界。
3. 集成测试：验证主链路能否打通。
4. 回归测试：验证新增能力不破坏现有电力工程能力。
5. 样例数据验收测试：验证固定样例在版本间行为稳定。
6. 非功能测试：验证性能、稳定性、错误提示和日志可读性。

建议每个阶段都至少覆盖三类样例：

1. `成功样例`
   验证主链路可跑通。
2. `边界样例`
   验证空字段、缺项、重复引用、未知类等场景。
3. `失败样例`
   验证错误可被识别、记录并对 UI 给出反馈。

## 6. 分阶段测试标准

### 6.1 P0 测试标准

#### 单元测试

- `CimObject` 属性、引用、Profile 标签读写正确
- `CimModel` 能按 `mRID`、类名、来源文件查询
- `CimIdIndex` 能建立对象与图元双向索引
- `CimGraphicMapper` 能把核心设备映射为正确图元类型

#### 组件测试

- 导入器能读取最小 `EQ + SSH` 样例
- 导入器能识别目录缺失、文件缺失、空文件
- 导入器能输出对象统计和错误信息
- 映射器能为核心设备生成可显示图元

#### 集成测试

- 从导入到对象树展示链路可跑通
- 从导入到单线图初稿生成链路可跑通
- 对象到图元、图元到对象双向定位可用
- 导入失败样例能在 UI 中给出明确反馈

#### 非功能测试

- 最小样例导入时间可接受，重复导入不崩溃
- 导入日志按对象数、错误数、警告数分组清晰
- UI 在空模型、失败模型下不出现卡死或无限弹窗

#### 通过标准

1. 关键单元测试全部通过。
2. 最小样例导入成功且对象数、设备类覆盖符合预期。
3. UI 中能看到对象树、图纸初稿和定位结果。
4. 导入异常时有可读错误信息，而不是静默失败。
5. 至少一条自动化 UI 用例覆盖导入与定位主链路。

### 6.2 P1 测试标准

#### 单元测试

- `CimValidationEngine` 能输出错误、警告和建议
- `CimTopologyMapper` 能生成稳定的拓扑快照
- `ICimExporter` 关键实现能导出最小对象子集

#### 组件测试

- 校验引擎能识别缺失引用、非法结构、未映射对象
- 拓扑映射能识别端子和连接关系
- 导出器能输出导出报告
- 回写模块能把关键图元属性同步到 `CimModel`

#### 集成测试

- 导入模型后可运行结构校验
- 导入模型后可运行导通、供电路径、接地路径分析
- 修改图元后可回写关键语义字段
- 最小导出子集可被重新导入或用于结果核对

#### 非功能测试

- 校验问题数量较多时列表可稳定加载和筛选
- 分析任务失败时错误信息能关联具体对象或步骤
- 导出报告字段完整，适合人工核对

#### 通过标准

1. 校验、分析、回写、导出四条链路全部可运行。
2. 分析结果与当前本地拓扑能力在关键样例上结论一致。
3. 回写后模型状态能通过基础校验。
4. 导出结果附带报告，能说明导出了什么、丢了什么。
5. 至少一组样例完成“导入 -> 校验 -> 分析 -> 回写 -> 导出”串行验证。

### 6.3 P2 测试标准

#### 单元测试

- 差异比较服务能识别新增、删除、修改
- 批处理服务能稳定组织任务输入输出
- 分析适配器接口满足统一协议

#### 组件测试

- 批量导入多个数据集不会互相污染状态
- 批量校验能输出汇总报告
- 差异报告格式稳定、字段清晰
- 适配器异常能被统一捕获并记录

#### 集成测试

- 多组模型可连续批量处理
- 两版模型可输出差异清单
- 统一适配器入口可接入内置和外部分析
- 批处理结果可回收为报告和日志

#### 非功能测试

- 连续批量执行时内存与临时文件不持续膨胀
- 失败任务不会阻断后续任务收尾
- 汇总报告可按任务、样例、错误等级追踪

#### 通过标准

1. 至少一组批量场景可稳定重复执行。
2. 差异比较结果与人工核对一致。
3. 统一分析适配接口在成功和失败场景都口径一致。
4. 批处理失败不影响其他任务收尾和结果归档。
5. 批处理报告能独立作为阶段验收附件使用。

### 6.4 P3 测试标准

#### 单元测试

- `DL / GL` 相关对象可正确解析

#### 组件测试

- 布局与地理信息导入导出前后一致

#### 集成测试

- 可处理包含 `DL / GL` 的数据集

- 外部工具链可稳定连接和回传结果

#### 非功能测试

- 大模型首屏加载、搜索、定位时间达到约定阈值
- 缓存重建或后端短时不可用时系统可恢复
- 发布包、升级包与兼容说明一致

#### 通过标准

1. 布局和地理数据可读、可写、可展示。
2. 大模型下浏览和查询性能达到预期阈值。
3. 外部工具链接入稳定，异常场景可恢复。
