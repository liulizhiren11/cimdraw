# CIM 标准知识体系

最后整理日期：2026-06-23

## 1. 文档目标

本目录不是项目设计文档，而是一套面向通用读者的 `CIM / CGMES` 学习资料。

目标有三点：

1. 让没有电力信息模型背景的读者建立完整心智模型。
2. 让已经接触过设备台账、图纸、拓扑、量测或模型交换的人，能够把零散概念连成体系。
3. 让读者读完后，能够理解 `CIM` 在标准、建模、交换、拓扑、图形和工程落地上的基本逻辑。

这套文档尽量做到“从零可读”，但不追求把 `IEC` 原文逐页复述，而是优先建立结构化理解。

## 3. 目录结构

### 3.1 认知基础

1. `01`
   解释 `CIM` 为什么存在，它在标准家族中的位置是什么。
2. `02`
   解释 `CIM` 如何建模，为什么 `UML`、类、属性、关联、多重性很重要。
3. `03`
   解释 `CIM` 的核心对象世界观，以及设备、端子、连接点、拓扑节点之间的关系。

### 3.2 交换与分析

1. `04`
   解释 `Profile`、`RDF/XML`、跨文件引用与交换头。
2. `05`
   解释连接性、开关状态、拓扑重建和导入拓扑的关系。
3. `06`
   解释图形布局、地理、量测、状态变量和运行数据。

### 3.3 工程进阶

1. `07`
   解释 `CGMES`、边界模型、差异模型、版本治理和工程落地场景。
2. `08`
   汇总高频误区、术语表和建议学习路径，帮助读者形成长期学习框架。

## 4. 与项目设计文档的关系

本目录回答的是“标准知识是什么”。

`docs/design/` 回答的是“`lz-power` 准备如何承接这些知识并落地实现”。

两者关系如下：

1. 本目录优先建立通用标准认知。
2. `design/` 目录把通用认知映射到当前项目结构、功能阶段和实现边界。
3. 如果读者已经具备 `CIM` 基础，进入 `design/` 会更容易理解项目为什么选择 `EQ / SSH / TP / SV` 为优先范围。

## 5. 使用方式

如果你是第一次接触 `CIM`，不要一开始就背标准编号或类名清单。

更有效的顺序是：

1. 先理解问题域。
2. 再理解建模方法。
3. 再理解对象关系。
4. 再理解 `Profile` 和文件交换。
5. 最后再进入拓扑、量测、图形和工程治理。

这是因为 `CIM` 的真正难点不是名词多，而是层次多、视角多、同一个对象会在不同语境下出现不同角色。
