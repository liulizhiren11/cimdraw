# 04. Profile 机制与 RDF 交换

## 1. 为什么一张网不会只用一个文件表达

很多初学者第一次看到 `CGMES` 或 `CIM XML` 会疑惑：

1. 为什么同一张网会拆成多个文件。
2. 为什么有 `EQ`、`SSH`、`TP`、`SV`、`DL`、`GL` 这些不同 `Profile`。
3. 为什么一个对象能在多个文件中间接参与同一张语义网络。

答案很简单：

因为电网模型里的信息天然分层，不同信息更新频率、责任边界和使用场景不同。

如果把一切都塞进一个超级文件：

1. 不利于分工交换。
2. 不利于版本治理。
3. 不利于按需导入。
4. 不利于解释“哪些是静态事实，哪些是运行断面”。

所以 `Profile` 机制是 `CIM/CGMES` 的核心设计，不是附属细节。

## 2. 什么是 Profile

可以把 `Profile` 理解为：

在同一个 `CIM` 语义世界中，针对某一类用途裁剪出来的一组类、属性和交换规则。

它不是另一个标准，而是同一套语义骨架下的“用途视图”。

## 3. 常见 Profile 及其作用

### 3.1 EQ

`EQ` 通常表示设备与静态结构相关内容。

它主要回答：

1. 网里有哪些设备。
2. 设备之间在静态接线层面如何组织。
3. 设备的静态属性是什么。

如果没有 `EQ`，后续很多内容都会失去附着基础。

### 3.2 SSH

`SSH` 常与稳态假设或运行断面属性相关。

它主要回答：

1. 当前断面下开关状态如何。
2. 某些运行相关设定值是什么。

`SSH` 往往不会重新定义整个设备世界，而是站在已有设备语义上补充当前断面状态。

### 3.3 TP

`TP` 常表示拓扑结果层。

它更偏“考虑开关状态后形成的拓扑节点和关系”。

它的重要意义在于：

1. 可以直接交换已有拓扑结果。
2. 可以与本地重建拓扑互相校验。

### 3.4 SV

`SV` 常表示状态变量结果层。

它更偏计算结果或状态估计结果，如：

1. 电压
2. 潮流
3. 开关状态结果

### 3.5 DL

`DL` 与图形布局相关。

它回答的不是“这个设备在电气上怎么连”，而是“图上怎样摆、怎样画、怎样定位”。

### 3.6 GL

`GL` 与地理信息相关。

它更适合地理视图、地图视图或空间位置相关场景。

## 4. 为什么 Profile 之间必须互相关联

单独一个 `Profile` 往往不足以支持完整业务。

例如：

1. 只有 `EQ`，你知道设备存在，但不知道当前开关断面。
2. 只有 `SSH`，你无法理解状态是附着在哪些设备上。
3. 只有 `TP`，你能看到拓扑节点，但很难解释它们对应哪些静态接线对象。
4. 只有 `DL`，你可能看到图形位置，但不一定知道语义设备是什么。

所以工程实现里，关键不是“支持某个文件”，而是“正确处理多个 `Profile` 的语义拼接关系”。

## 5. RDF/XML 为什么常见

很多人接触 `CIM` 时，首先看见的是 `XML`。

但更准确地说，常见交换格式是基于 `RDF/XML` 的表达方式。

这意味着文件不只是普通树状 XML，而更像在用 XML 语法承载一张对象关系图。

## 6. 三个最重要的 RDF 机制

### 6.1 rdf:ID 或 rdf:about

它们用于标识对象。

理解重点：

1. 对象必须可被引用。
2. 交换时需要稳定标识。
3. 一个文件中的对象，可能被另一个文件引用。

### 6.2 rdf:resource

它用于表达对象之间的引用关系。

例如：

1. 一个 `Terminal` 引用某个 `ConductingEquipment`
2. 一个 `Terminal` 引用某个 `ConnectivityNode`

所以看到 `rdf:resource` 时，不要把它当普通文本字段，要把它理解成“语义链接”。

### 6.3 命名空间

命名空间用于区分不同语义域和标签来源。

它的意义在于：

1. 避免重名冲突。
2. 明确类和属性来自哪个标准语境。
3. 支持模型头、元信息和主体对象共存。

## 7. 一个重要认知：文件是分开的，语义网络是一体的

这句话非常关键。

在 `CIM/CGMES` 中：

1. 文件可以拆开。
2. 对象可以分布。
3. 引用可以跨文件。
4. 但语义上它们仍属于同一张模型网络。

因此，导入器真正要做的事不是“顺序读几个 XML”，而是：

1. 识别对象
2. 归并对象
3. 解析引用
4. 拼接关系
5. 校验结构

## 8. 交换头为什么重要

`Model` 描述头常被初学者忽略，但它对工程非常关键。

它通常会告诉你：

1. 当前文件属于哪个 `Profile`
2. 文件依赖哪些模型
3. 版本和 URI 信息是什么

这些信息直接影响：

1. 导入顺序
2. 依赖关系校验
3. 跨文件拼装
4. 版本解释

## 9. 工程上应该怎样理解导入顺序

虽然具体实现未必必须线性固定，但概念上可以这样理解：

1. 先有 `EQ`
2. 再叠加 `SSH`
3. 再对照 `TP`
4. 再补充 `SV`
5. 最后再考虑 `DL / GL`

这个顺序不是死规则，但很符合初学者建立认知的路径，也符合很多工程导入器的最小闭环思路。

## 10. 一句话理解本章

`Profile` 机制是在同一语义世界里按用途拆分模型，`RDF/XML` 则是把这张跨文件对象关系网表达出来的常见交换载体。

## 11. 读完本章后你应该具备的认知

1. 知道为什么一张网会拆成多个 `Profile` 文件。
2. 知道 `EQ / SSH / TP / SV / DL / GL` 各自大致负责什么。
3. 知道 `RDF/XML` 的重点不是 XML 树，而是对象引用网络。
4. 知道导入的本质是对象归并和关系拼接，而不是逐文件顺序读文本。
