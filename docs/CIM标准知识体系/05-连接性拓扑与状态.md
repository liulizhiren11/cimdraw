# 05. 连接性、拓扑与状态

## 1. 这是最容易“懂一半”的章节

很多人会说自己懂拓扑，因为知道什么是节点、支路、开关。

但在 `CIM` 里，真正难的是把下面三层清楚分开：

1. 静态接线结构
2. 开关状态
3. 运行拓扑结果

如果这三层混了，后续所有分析都会变得含糊。

## 2. 静态连接层：谁接在谁上面

这一层主要由 `Terminal` 和 `ConnectivityNode` 支撑。

### 2.1 Terminal

它把设备接入网络。

### 2.2 ConnectivityNode

它表示静态连接点。

因此，静态连接层回答的是：

1. 设备有哪些连接端。
2. 每个端接到哪个连接点。
3. 图纸或设备结构层面的接线关系是什么。

这一层不直接回答“现在导不导通”。

## 3. 状态层：哪些设备当前开断

这一层主要通过开关类对象及其当前状态表达。

例如：

1. 断路器当前合上还是断开。
2. 隔离开关当前合上还是断开。
3. 某些运行断面设定值是什么。

状态层回答的是：

1. 静态可连接结构在当前运行时刻如何被激活或切断。

## 4. 拓扑层：最终形成了哪些连通团

考虑状态后，系统可以从静态连接结构推导出拓扑结果。

这时就会出现：

1. `TopologicalNode`
2. 可能还有更高层的拓扑岛或连通分区概念

这一层回答的是：

1. 当前运行条件下，哪些导电部分属于同一个拓扑节点。
2. 网络被切成了几个连通部分。
3. 某条供电路径是否存在。

## 5. 为什么 ConnectivityNode 不等于 TopologicalNode

这是整章最重要的一句话。

### 5.1 ConnectivityNode

描述静态接线事实。

### 5.2 TopologicalNode

描述考虑当前开关状态后的运行连通结果。

同一个 `ConnectivityNode` 集合，在不同开关状态下，可能形成不同的 `TopologicalNode` 组合。

反过来，多个静态连接点也可能在某些运行状态下被归入同一个拓扑节点团。

## 6. 导入拓扑和本地重建拓扑为什么可能不一致

这是工程里非常常见的现象，不一定代表谁错了。

原因可能包括：

1. 外部系统导出的 `TP` 基于某个特定断面。
2. 本地系统使用的 `SSH` 不是同一时间断面。
3. 某些开关或特殊连接设备被不同系统采用了不同建模口径。
4. 本地重建时忽略了边界、虚拟连接或特殊设备规则。

所以成熟系统不会简单假设“导入的 `TP` 一定等于本地拓扑重建结果”，而是会：

1. 明确两者来源
2. 对比差异
3. 给出解释

## 7. 节点-开关模型与母线-支路模型

很多读者会带着传统分析模型经验来看 `CIM`。

这没有问题，但要意识到：

1. `CIM` 常常更接近节点-开关或设备-端子的表达语义。
2. 具体分析引擎可能更偏母线-支路或其他计算模型。

这意味着在工程实践中，往往需要一个中间转换过程：

1. 从 `CIM` 语义模型出发
2. 解析设备、端子、连接点和状态
3. 生成分析所需的网络模型

所以 `CIM` 不等于分析内核输入格式，但它可以成为分析模型的重要上游语义源。

## 8. 路径分析为什么离不开拓扑层

无论是：

1. 导通分析
2. 供电路径分析
3. 接地路径分析
4. 分岛识别

它们本质上都不是仅靠“图上谁挨着谁”就能做出来的。

它们依赖的是：

1. 稳定的连接对象语义
2. 明确的状态输入
3. 可解释的拓扑重建规则

## 9. 一个实用判断法

当你看到某个“连接”概念时，可以先问它属于哪一层：

1. 如果回答“图纸上接在一起”，多半偏静态连接层。
2. 如果回答“当前开关是否闭合”，偏状态层。
3. 如果回答“现在是否属于同一电气连通团”，偏拓扑层。

只要能把问题先归层，很多混乱都会消失。

## 10. 工程上应保留的三种结果

成熟实现通常会保留：

1. 原始静态连接关系
2. 当前状态输入
3. 计算或导入得到的拓扑结果

这样做的好处是：

1. 可追踪
2. 可解释
3. 可回放
4. 可比对

## 11. 一句话理解本章

`CIM` 中的拓扑不是图纸连线本身，而是“静态连接关系 + 当前状态”共同推导出的运行连通结果。

## 12. 读完本章后你应该具备的认知

1. 知道静态连接、状态、拓扑结果是三层不同语义。
2. 知道 `ConnectivityNode` 和 `TopologicalNode` 的根本区别。
3. 知道 `TP` 导入结果和本地重建结果可能不一致，而且需要可解释。
4. 知道真正的分析能力依赖的是可追踪的拓扑语义，而不是单纯画线关系。
