# 06. 图形布局、量测与运行数据

## 1. 很多系统的问题不在模型本身，而在“把不同视图混成一个视图”

电力软件里常见三种内容：

1. 设备和连接语义
2. 图形和布局表达
3. 量测与运行结果

如果这三种内容混在一起，系统会很快变得难维护。

`CIM` 的一个重要价值，就是帮助我们把这些层次拆开，同时又保持关联。

## 2. 图形视图不等于语义模型

这是初学者最容易忽略的一点。

图上看起来相邻，不一定语义上连接。

反过来，语义上连接，也不一定必须画成某一种图形布局。

因此在 `CIM` 的世界里，通常要分开看：

1. 语义对象
2. 图形对象
3. 图形对象与语义对象的映射

## 3. DiagramLayout 在回答什么问题

`DL` 相关信息主要回答：

1. 某个语义对象在图上如何摆放。
2. 画布坐标是什么。
3. 某条图形连接线如何表现。
4. 文本、符号、可视层可能如何组织。

它回答的不是：

1. 这个设备是不是导通。
2. 这个对象属于哪个拓扑节点。
3. 它的实时电压电流是多少。

## 4. 为什么同一份语义模型可以有多种图形视图

这是理解图模分离的关键。

同一套设备和连接语义，可以有：

1. 调度单线图
2. 配电一次图
3. 教学示意图
4. 运维查看图

这些图的布局、符号、密度都可能不同，但它们指向的语义设备可以是同一套对象。

因此，图形层应被理解为“视图层”，而不是“电气事实层”。

## 5. 地理信息与图形布局不是一回事

`GL` 与 `DL` 常被一起提起，但它们不是同一层：

1. `DL`
   更偏画面布局、单线图坐标、示意图表达。
2. `GL`
   更偏真实空间位置、地理坐标、地图语义。

同一个站内开关在单线图里可能画得非常紧凑，但在地理视图里会分布在真实空间位置上。

## 6. 量测对象在 CIM 中的角色

量测层回答的不是“设备是什么”，而是“设备当前被观测到了什么”。

典型例子包括：

1. 电压量测
2. 电流量测
3. 功率量测
4. 开关位置信号

要点在于：

1. 量测是观测点，不是设备本体。
2. 一个设备可以对应多个量测点。
3. 一个量测点有自身的类型、单位、质量和来源语义。

## 7. 状态变量与量测的区别

很多初学者会把量测值和状态变量混在一起。

这两者相关，但不等同。

### 7.1 量测

更偏原始观测或外部采样输入。

### 7.2 状态变量

更偏某个分析、估计或运行结果模型中的当前状态表达。

例如：

1. `SvVoltage`
2. `SvPowerFlow`
3. `SvStatus`

这些对象往往更像“结果层对象”，而不是“原始设备层对象”。

## 8. 为什么运行数据不能直接改写设备定义

一个成熟模型必须区分：

1. 设备是什么
2. 设备现在怎么样

如果把运行态直接写回静态定义，会出现很多问题：

1. 无法区分主数据和断面数据
2. 无法保留历史
3. 无法支持多时刻、多场景并存

因此工程上通常会把：

1. 静态设备模型
2. 运行断面模型
3. 分析结果模型

拆开管理。

## 9. 图模一体化的正确理解

“图模一体化”并不意味着图形对象和语义对象必须合并成一个类。

更成熟的理解是：

1. 图形和模型是两层。
2. 两层之间有稳定映射。
3. 用户在图上选中设备，能定位到语义对象。
4. 语义对象变化后，图上能同步更新。

这样才既保留可视交互，又不破坏语义清晰性。

## 10. 一个实用框架

看到系统中的某个元素时，可以问：

1. 它是设备对象吗。
2. 它是图形对象吗。
3. 它是量测点吗。
4. 它是状态结果吗。
5. 它只是这些对象之间的映射关系吗。

只要能回答这五个问题，系统分层通常就不会太乱。

## 11. 一句话理解本章

图形、地理、量测和状态都很重要，但它们不是电网语义模型本体，而是围绕语义模型展开的不同视图和结果层。

## 12. 读完本章后你应该具备的认知

1. 知道图形视图不等于电气事实。
2. 知道 `DL` 和 `GL` 的边界。
3. 知道量测、状态变量和静态设备不是同一类对象。
4. 知道图模一体化的关键是映射稳定，而不是强行把所有内容揉成一层。
