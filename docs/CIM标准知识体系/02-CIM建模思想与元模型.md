# 02. CIM 建模思想与元模型

## 1. 为什么先学建模思想

很多人学 `CIM` 时喜欢直接背类名，例如：

1. `Breaker`
2. `ACLineSegment`
3. `PowerTransformer`
4. `Terminal`

这种方式短期能记住一些名词，但很快就会遇到瓶颈，因为 `CIM` 的真正难点不是“类很多”，而是“这些类为什么要这样组织”。

所以本章的重点不是背清单，而是建立三个能力：

1. 看懂 `CIM` 为什么大量使用继承。
2. 看懂属性和关联的角色差异。
3. 看懂一个对象到底属于设备层、连接层、拓扑层还是运行层。

## 2. CIM 的建模立场

`CIM` 并不是把现实世界一比一复制成软件对象，而是带着明确目的去抽象电力系统。

这个目的主要有三类：

1. 描述电力系统中的对象。
2. 描述对象之间可交换、可计算、可追踪的关系。
3. 支持不同业务系统之间共享同一种语义。

因此，`CIM` 的建模不是“谁长得像谁就归一类”，而是“谁在电网语义中承担同类角色就归一类”。

## 3. UML 在 CIM 中的作用

`CIM` 大量通过 `UML` 元模型来定义类、属性和关联。

你不需要成为 `UML` 专家，但至少要懂下面这些概念：

1. `Class`
   描述一种对象类型。
2. `Attribute`
   描述对象自身的值。
3. `Association`
   描述对象与对象之间的关系。
4. `Inheritance`
   描述父类到子类的抽象层次。
5. `Multiplicity`
   描述关系的数量约束。
6. `Enumeration`
   描述有限取值集合。

没有这些概念，后面看 `CIM` 文件时会只看到标签，看不到结构。

## 4. 属性和关联不是一回事

这是学习 `CIM` 的第一个关键分界。

### 4.1 属性

属性是对象自己的值。

例如：

1. 名称
2. 额定电压
3. 开关常开常闭标识
4. 有功、无功设定值

### 4.2 关联

关联是对象和对象之间的关系。

例如：

1. 一个 `Terminal` 属于哪个设备。
2. 一个 `Terminal` 接到哪个 `ConnectivityNode`。
3. 一个 `VoltageLevel` 属于哪个 `Substation`。

为什么要强调这个区别？

因为很多系统在做表结构时，喜欢把一切都看成字段。但在 `CIM` 里，一个对象的重要性往往不只来自它自己的数值，而是来自它和其他对象的连接关系。

## 5. 继承为什么这么多

初学者经常觉得 `CIM` 的继承树太深。

例如一个看起来很具体的设备，往上可能会经过：

`IdentifiedObject -> PowerSystemResource -> Equipment -> ConductingEquipment -> Switch -> Breaker`

这并不是为了“面向对象而面向对象”，而是因为每一层都承担了不同抽象角色。

### 5.1 IdentifiedObject

这是很多对象的共同基础，主要解决“我是谁”的问题。

常见内容包括：

1. `mRID`
2. `name`
3. `description`

### 5.2 PowerSystemResource

这一层开始进入电力系统资源语义。

它强调这个对象属于电力系统资源，而不是一个纯粹的通用对象。

### 5.3 Equipment

这一层说明它是设备或可管理设备单元的一部分。

### 5.4 ConductingEquipment

这一层说明它是导电设备，会参与电气连接。

不是所有设备都会导电，因此这一层很关键。

### 5.5 具体设备类

例如：

1. `Breaker`
2. `Disconnector`
3. `PowerTransformer`
4. `EnergyConsumer`

这一层才进入具体专业含义。

## 6. 多重性为什么重要

`Multiplicity` 看起来很像细节，实际上非常关键。

例如：

1. 一个设备通常可以有多个 `Terminal`。
2. 一个 `Terminal` 在某个时刻通常连接到一个 `ConnectivityNode`。
3. 一个 `Substation` 可以包含多个 `VoltageLevel`。

这些约束不仅影响理解，也直接影响：

1. 文件合法性校验
2. 数据库建模
3. UI 对象树组织
4. 拓扑重建算法

如果不理解多重性，后面做模型校验时就很容易把“结构错误”误当成“内容缺失”。

## 7. 抽象类的价值

在 `CIM` 里，很多类的存在不是为了被直接实例化，而是为了统一语义口径。

例如：

1. 你可能不会直接看到某些抽象父类对象出现在交换文件中。
2. 但这些父类定义了子类共享的结构和约束。

这能带来两个好处：

1. 标准更一致。
2. 软件实现时可以按层复用逻辑。

## 8. 单位、量纲与枚举

`CIM` 不只是“有个数值”这么简单，它还很重视数值的语义背景。

所以在理解属性时，要同时关注：

1. 数值本身
2. 单位
3. 倍率
4. 枚举类型

例如一个值如果没有明确单位，软件之间很容易在 `kV / V`、`MW / kW`、`A / kA` 上产生错解。

## 9. 元模型视角下，CIM 比数据库表更像“语义骨架”

如果把 `CIM` 误解为“数据库表模板”，就会出现两个常见问题：

1. 只关注字段，不关注对象关系。
2. 只关心存储，不关心交换和解释。

更合适的理解是：

1. `CIM` 定义了对象语义骨架。
2. 数据库、XML、图数据库、内存对象，只是这个骨架的不同承载方式。

## 10. 读图的基本方法

后续不管你看 `UML` 图、类文档、工具生成代码还是 `RDF/XML`，都建议固定问五个问题：

1. 这个类是抽象层还是业务层。
2. 它的父类是谁，继承来什么共同语义。
3. 它有哪些关键属性。
4. 它和哪些对象存在关键关联。
5. 它在静态设备、拓扑、量测、图形、状态中的角色是什么。

只要坚持用这五个问题去看类，学习速度会明显快很多。

## 11. 一句话理解本章

`CIM` 的核心不在于背类名，而在于理解它如何用 `UML` 把电力世界拆成层次清晰、语义稳定、可交换的对象网络。

## 12. 读完本章后你应该具备的认知

1. 知道属性和关联的区别。
2. 知道继承树不是装饰，而是在表达抽象层次。
3. 知道多重性、抽象类、单位和枚举会直接影响工程实现。
4. 知道后续看到任何 `CIM` 类，都应先问它在语义骨架中的位置。
