# 01. CIM 是什么与标准地图

## 1. 先回答一个根问题

`CIM` 的全称是 `Common Information Model`，中文常译为“公共信息模型”或“通用信息模型”。

它不是某一个软件，不是某一种数据库，也不是某一种图纸格式。

它更像一套面向电力系统的统一语义语言，用来回答下面这些问题：

1. 一个变压器、开关、母线、负荷、端子、连接点，在不同系统里应该如何被一致地描述。
2. 一个系统导出的模型，另一个系统怎样才能读懂，不至于只看见字段、看不见含义。
3. 设备静态信息、运行断面信息、拓扑信息、状态变量、图形布局信息，怎样既分开管理，又能互相关联。

如果把电网看成一个复杂世界，那么 `CIM` 的作用不是“画图”，而是定义这个世界的对象、属性、关系和交换方式。

## 2. 没有 CIM 时会发生什么

在很多电力信息系统中，同一个真实设备往往会被不同系统用不同方式表示：

1. 调度系统更关心运行拓扑和状态量。
2. 资产系统更关心设备台账、厂家、寿命和运维信息。
3. GIS 更关心地理位置。
4. 图形系统更关心画面符号、坐标和连线。
5. 分析软件更关心导纳、参数、边界条件和计算结果。

如果没有统一语义层，这些系统就会出现典型问题：

1. 同一设备在不同系统里有不同名字或不同主键。
2. “连接关系”在一个系统里是端子关系，在另一个系统里是图元贴靠关系，无法直接互认。
3. 拓扑节点来自某个外部计算结果，但本地系统只保存静态设备，解释不清来源。
4. 一份模型导出后进入下游系统，字段都在，语义却已经偏了。

所以 `CIM` 的价值，不是让所有系统长得一样，而是让它们在交换时说同一种语言。

## 3. CIM 解决的不是单个软件问题，而是系统协同问题

很多初学者第一次接触 `CIM` 时，会误以为它只是“某个软件支持导入的一种 XML”。

这会把理解范围缩得太小。

更准确的说法是：

1. `CIM` 首先是领域语义标准。
2. 其次才是模型交换标准。
3. 最后才会在具体软件中表现为文件、数据库、接口和对象类。

因此，学习 `CIM` 的正确顺序不是先问“这个标签是什么意思”，而是先问：

1. 这个标准想统一的对象到底是什么。
2. 为什么它要把设备、端子、连接点、拓扑节点、状态变量拆开。
3. 为什么同一张网会被拆分成多个 `Profile` 文件交换。

## 4. 标准家族地图

### 4.1 IEC 61970

`IEC 61970` 最初主要面向 `EMS` 与输电网相关的应用集成。

它常被理解为“偏输电、偏调度、偏模型交换和分析”的一支。

其中最常被提到的内容包括：

1. `61970-301`
   偏核心模型语义，常被视作理解 CIM 类体系的重要入口。
2. `61970-452`
   常用于静态网络模型交换相关场景。
3. `61970-456`
   常与稳态状态假设相关。
4. `61970-552`
   非常关键，定义了 `CIM XML Model Exchange Format`，也就是很多人日常接触到的 `RDF/XML` 交换格式基础。

### 4.2 IEC 61968

`IEC 61968` 更偏企业级应用集成，尤其常与配电、企业信息交换和业务系统协作相关。

如果粗略理解：

1. `61970` 更偏模型和运行分析。
2. `61968` 更偏企业业务集成和配电应用。

但两者并不是互斥世界，而是共享同一个 `CIM` 语义基础，在不同业务边界上展开。

### 4.3 IEC 62325

`IEC 62325` 更多与电力市场通信有关。

它和 `CIM` 也有关联，但对刚入门的读者来说，不必一开始就深入。前期优先把 `61970 / 61968 / CGMES` 建起来更重要。

## 5. 什么是 CGMES

`CGMES` 的全称是 `Common Grid Model Exchange Specification`。

可以把它理解为：在 `CIM` 总体语义框架下，为电网模型交换形成的一套更具体、更可操作、工程界更常落地的规范体系。

它的重要性在于：

1. 它把“CIM 的语义类库”进一步落实为“模型交换时应该怎样拆分、怎样组织、怎样命名、怎样互相引用”。
2. 它在跨区域、跨组织电网模型交换中具有很强的工程影响力。
3. 很多开源工具、样例数据、行业讨论，实际谈的并不是抽象的 `CIM` 全貌，而是 `CGMES` 的一组 `Profile` 与文件组织方式。

因此，学习时要始终记住：

1. `CIM` 是更大的语义世界。
2. `CGMES` 是在这个语义世界中非常重要的一种工程化交换落地方式。

## 6. 学习 CIM 时最容易混淆的三层

### 6.1 语义层

这一层回答“世界里有哪些对象，它们是什么关系”。

例如：

1. `Breaker` 是一种开关设备。
2. `Terminal` 是设备的导电连接端。
3. `ConnectivityNode` 表示静态连接关系。

### 6.2 表达层

这一层回答“这些对象被怎样写进文件、数据库或接口里”。

例如：

1. 在 `RDF/XML` 中使用 `rdf:ID` 或 `rdf:about` 标识对象。
2. 使用 `rdf:resource` 表达对象间引用。
3. 用命名空间区分类和属性。

### 6.3 应用层

这一层回答“软件拿到模型后拿它做什么”。

例如：

1. 浏览对象树。
2. 绘制单线图。
3. 重建拓扑。
4. 跑潮流、状态估计、短路分析。
5. 比较两版模型差异。

很多误解都来自把这三层混在一起。

## 7. 一句话理解本章

如果只记住一句话，可以记成：

`CIM` 不是一种文件，而是一套让电力系统对象能够被一致理解、表达和交换的公共语义体系；`CGMES` 则是这套体系在电网模型交换上的重要工程化落地。

## 8. 读完本章后你应该具备的认知

1. 知道 `CIM` 首先是语义模型，不是文件格式。
2. 知道 `IEC 61970`、`IEC 61968`、`CGMES` 之间的大致边界。
3. 知道学习 `CIM` 时要区分语义层、表达层和应用层。
4. 知道后续学习不应该陷入标准编号堆里，而应该先建立对象世界观。
